Components

Link

Link
Markup <a href="#">Link</a>
  • Links go places, while buttons do things. The difference is that buttons change the state of the website. That can be on the server side or the client side, but a button implies that some action is being taken, and something, somewhere is different now. Links take you somewhere else, but neither the place you were nor the place you’re going have changed when you do so.
  • Yes, buttons are routinely abused to work as just bigger, more prominent links — usually along with other bad design decisions that visually mark almost all of the items on a page as the most important thing, creating frustration and increasing the user’s cognitive load. It’s a terrible practice, and just because it’s common is no reason to do it.
  • In places where you might be tempted to use a button to go somewhere because the link should be visually prominent, consider using the prominent link component instead. Just remember: if everything is visually prominent, then nothing is.
  • If you need to present the user with a prominent decision point when she could go to any of several different places, and each one calls for some significant explanation, you might consider the panel link component.

Call to Action

Call to Action
Markup <a href="#" class="cta">Call to Action</a>
  • Use this component when you want to give users the opportunity to go somewhere (meaning that they’re not changing the website back-end or front-end), but the link should be visually prominent.
  • This component is suitable as a call to action where the action is to go somewhere, rather than to do something. As with other calls to action, you should not have more than one on a page. The user can only do one thing; if you ask hen to do several things, you only create confusion and frustration.
  • Like any other option that creates visual prominence, do not overuse this component. Generally speaking, you should not use this more than once on a page. Visual hierarchy helps create clarity, unless you try to make everything prominent, then there is no hierarchy and no clarity, because everything is equally prominent. It’s the design equivalent of writing in all caps or shouting at someone.
  • If the call to action does do something (changing the state of the website, either on the front-end or the back-end), then you should use a button instead.

    Text Input

    Markup <input type="text" placeholder="Text input"> <input type="email" placeholder="you@example.com"> <input type="search" placeholder="Your search terms..."> <input type="tel" placeholder="1 (212) 867-5309"> <input type="url" placeholder="https://example.com"> <input type="number" value="0" min="0" max="10"> <input type="text" placeholder="Text input" value="Error!" class="error">
    • Do not use placeholders in place of labels. When you begin typing in a text box, the text disappears, so if you forget precisely what you need to enter in this box, you’ll need to remove what you’ve written. Placeholders aren’t designed to take the place of labels; that’s what labels are for. Placeholders should be used to provide an example of the sort of input a user should provide in the box.
    • The text input is the most generic of the fields provided here. This is appropriate for fields that need to collect a piece of arbitrary short text. Wherever appropriate, use one of the more specific fields detailed here. The text input is the default option that you can use when none of the other options quite fit. When asking the user for text that you expect to exceed more than a dozen or so characters, use the text area component instead.
    • The email input is the field you should use when you ask the user to enter an email address. Though this may look the same on a desktop browser, the specialized field can prompt a variety of affordances. For example, on iOS Safari, the email field brings up a different keyboard, with special @ and .com buttons that make typing an email address more convenient.
    • The search input is the field you should use when you ask the user to enter a search query.
    • The telephone input is the field you should use if you ever need to ask the user to enter a telephone number.
    • The URL input is the field you should use when you ask the user to enter a URL.
    • The number input is the field you should use when you ask the user to enter a number.
    • Never ask for more data than you actually need. Every additional field on a form increases the interaction cost, reducing the likelihood that you’ll get any response at all. Interrogate every field to determine if it’s something you really need, and eliminate everything that is not absolutely necessary in order to provide the service that you wish to provide.

    • Do not use placeholders in place of labels. When you begin typing in a text box, the text disappears, so if you forget precisely what you need to enter in this box, you’ll need to remove what you’ve written. Placeholders aren’t designed to take the place of labels; that’s what labels are for. Placeholders should be used to provide an example of the sort of input a user should provide in the box.
    • It is critically important that error messages be written in as clear and concise a manner as possible. They should make it clear to the user, in approachable, everyday language, why the error occurred and what options the user has to rectify it.

    Text Area

    Markup <textarea class="sm">This is a small text area.</textarea> <textarea>This is the standard text area.</textarea> <textarea class="lg">This is a large text area.</textarea> <textarea maxlength="256">This is a standard text area with a maximum length of 256 characters.</textarea>
    • The text area input is appropriate when you’re asking a user to enter a significant amount of text (anything more than a dozen or so characters). For short bits of text, you can use a text input field.
    • Just like other form elements, text areas should appear with a concise and informative label.
    • If a text area has a maximum number of permitted characters, you can add it to the <textarea> tag with the maxlength attribute. In most browsers, this will make it impossible for users to enter illegal values. This will also prompt the character counter enhancement, as demonstrated above. Character limits should generally follow from the nature of the data being entered, rather than arbitrary cut-offs, which could stop users from entering full, complete answers.

    Input & Button

    Markup <fieldset class="input-button"> <input type="search" placeholder="Search"> <button>Search</button> </fieldset>
    • This component allows you to create a form that combines a single text input field with a button. This is useful for simple, single-input forms like search query forms and email list signup forms.

    Dropdown

    Markup <select> <optgroup label="Primary Colors"> <option value="red">Red</option> <option value="blue">Blue</option> <option value="green">Green</option> </optgroup> <optgroup label="Secondary Colors"> <option value="orange">Orange</option> <option value="yellow">Yellow</option> <option value="purple">Purple</option> </optgroup> </select>
    • Use a dropdown when you would like a user to choose a single option from a set list of options. When to use a dropdown and when to use a set of radio buttons can be subtle, but some reasons to prefer a dropdown over radio buttons include:
      • When there is a clear default, recommended, or most common option (which you should have selected by default).
      • There are quite a few options to choose from, but it’s easy for users to notice the pattern and predict what the available options will be (e.g., a list of countries).
      • When you have more than seven options to choose between.

    Radio Buttons

    Markup <label class="radio"> <input type="radio" name="radio-example" value="yes" id="radio-example-yes"> Yes </label> <label class="radio"> <input type="radio" name="radio-example" value="no" id="radio-example-no"> No </label> <label class="radio"> <input type="radio" name="radio-example" value="note" id="radio-example-note"> Well… <span class="note">This radio button’s label has a note.</span> </label> <label class="radio extended"> <input type="radio" name="radio-example" value="ext" id="radio-example-ext"> You may find… </label>
    • Use radio buttons when you would like a user to choose a single option from a set list of options. When to use radio buttons and when to use a dropdown can be subtle, but some reasons to prefer radio buttons over a dropdown include:
      • When there is no clear default, recommended, or most common option. When this is the case, do not select any option by default (though you may want to require that the user select an option on the back-end).
      • When it’s important that users be able to compare the options side-by-side.
      • When there are five or fewer options to choose from.

    Checkboxes

    Markup <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="designer" id="checkbox-example-designer"> Designer </label> <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="frontend-developer" id="checkbox-example-frontend-developer"> Frontend Developer </label> <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="backend-developer" id="checkbox-example-backend-developer"> Backend Developer </label> <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="fullstack-developer" id="checkbox-example-fullstack-developer"> Full Stack Developer </label> <label class="checkbox extended"> <input type="checkbox" name="checkbox-example" value="extended" id="checkbox-example-extended"> Actually... </label>
    • Use a set of checkboxes when you would like a user to choose one or more options from a set list of options.
    • You can also use a single checkbox on its own to capture a boolean value from a user, like opting in to an agreement.
    • If you use a checkbox to gather consent or agreement from a user, it cannot be checked by default. Consent is only ethical or binding when it is freely and consciously given. It is entirely too easy for a user to submit a form without noticing a checkbox that is marked by default. Besides being a deeply unethical dark pattern or ”antipattern,“ in many legal jurisdictions this will not stand up as legal consent.

    Block Quote

    We should work toward a universal linked information system, in which generality and portability are more important than fancy graphics techniques and complex extra facilities. The aim would be to allow a place to be found for any information or reference which one felt was important, and a way of finding it afterwards. The result should be sufficiently attractive to use that the information contained would grow past a critical threshold, so that the usefulness the scheme would in turn encouraged its increased use. The passing of this threshold accelerated by allowing large existing databases to be linked together and with new ones.

    Tim Berners Lee
    Markup <blockquote> <p>We should work...</p> <footer>Tim Berners Lee</footer> </blockquote>
    • The block quote component is for quoting large sections of text in a document. This is a block element which should include one or more <p>, <ul<, or <ol> tags.
    • A <footer> element can be included to provide attribution for the quote. When used in a longer text, this may not be necessary if you’ve already provided attribution in the text.

    Pull Quote

    “If we wish to get back to the business of quietly improving people’s lives, one thoughtful interaction at a time, we must rid ourselves of the cult of the complex.”

    Markup <aside class="pull"> <p>If we wish to get back...</p> </aside>
    • A pull quote is a short phrase or excerpt pulled from the main content and given greater prominence as a visual element. As such, we mark it semantically as an <aside> because it can be removed from the main flow of the the content without interrupting meaning. The pull class marks this aside for the visual rendering of a pull quote.

    Image

    Markup <img src="path/to/image.jpg" alt="Alt text" />
    • Consider if what you want to use might be a figure, rather than an unaccompanied image. While every image must include appropriate alternative text to make it accessible (see below), most images would likely benefit from a caption, which the figure component provides. It’s not impossible for an unaccompanied image to be the right choice, but it’s rare, so consider carefully if a figure could be a better choice.
    • Images must always include appropriate alternative text in the alt attribute. This text is read aloud to users using screen readers, so it should give a useful, succinct description of the image and the purpose that it plays on the page. You don’t need to specify that it is “an image of” or “a photo of”; that is understood from the context of being alternative text.
    • If you’re having a difficult time coming up with alternative text for an image, consider the possibility that the image might be purely decorative. This is usually a sign that you should not use the image. We might need to update one or more components in the design system to meet an aesthetic need, but images in content should be used because they serve a purpose. If you’re having a hard time describing that purpose for the image’s alternative text, that can be your first sign that the image doesn’t have one.

    • Embedded video can also be used in a figure to provide a caption.
